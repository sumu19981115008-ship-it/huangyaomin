# AutoBot 算法研究档案

本目录存放 AutoBot 自动打关机器人的各版本算法代码及研究成果，用于：
1. **关卡难度评定基准**：不同版本的 Bot 能通过的关卡集合反映了不同难度层次
2. **算法演化记录**：每个版本解决了什么问题、引入了什么新机制
3. **后续研究起点**：基于现有版本继续改进或开发更高级的前向搜索策略

## 算法版本演化

### v1 — 基础版（2026-04-22）

**核心逻辑：** 部署 buffer 或队列队首，无评分，先到先得。  
**问题：** 无可达性检测，大量无效车占满轨道导致死锁。  
**通关率：** A组 ~60%，B组 ~70%（估算）

---

### v2 — 弹药匹配评分（2026-04-22）

**核心逻辑：** `score = 1 / (1 + |弹药总和 - 方块数|)`，弹药最贴近方块数的颜色优先。  
**新增：** 可达性检测（四方向扫描外层），只部署可达色。  
**通关率：** A组 ~72%

---

### v3 — 严格可达性（2026-04-23）

**核心逻辑：** 可达性改为严格不穿透——每行/列只取最外层第一个非空格子，与 `_findTarget` 完全一致。  
**通关率：** A组 ~75%

---

### v4 — buffer 危险预判（2026-04-23）

**核心逻辑：** 检测 `buffer.length + 即将完圈车数 >= bufferCap - 1`，危险时强制清 buffer。  
**新增：** `_pruneUselessLanes` 解除队列前端无效颜色阻塞（修复 L33 死锁）。  
**通关率：** A组 ~82%，B组 ~86%

---

### v5 — 轨道颜色多样性惩罚（2026-04-24）

**核心逻辑：** 轨道上已有同色车则评分乘以 `0.6^n`，避免单色占满轨道。  
**通关率：** A组 86.6%（259/299），B组 90.1%（154/171）

---

### v6 — pathPos 曝光感知（2026-04-24）

**核心逻辑：** 新增 `computeColorExposurePathPos()`，按轨道路径顺序计算每种颜色的首次曝光 pathPos。兜底模式（全不可达）时，按曝光 pathPos 排序选最早会变可达的颜色。  
**改进：** 消除了所有"失败"结局，A/B 组失败数降至 0。  
**通关率：** A组 86.6%（259/299，**0失败**），B组 90.1%（154/171，**0失败**）

---

### v7 — 全队列弹药统计 + 解锁机制（2026-04-24）

**核心 bug 修复：** `colorAmmo` 原来只统计各队列队首弹药（~20发），导致所有颜色弹药匹配分趋同、评分无意义。改为扫描**全队列 + buffer** 后评分才真正有效。  
**新增：** `unlockPool` 机制——检测队列头部不可达但后续有可达色时，评估停车成本与收益。  
**效果：** A100 首次自动通过（原先因 `#00CED1` 阻塞 L0 而死锁）。  
**通关率：** A组 87.6%（262/299），B组 91.8%（157/171）

---

### v8 — 停车场策略 + commitLane（2026-04-24）

**核心逻辑：** allEmpty 挖坑——轨道为空且全不可达时，若某条队列 `gain > cost × 1.2 && dist ≤ 3`，主动停车挖队列。  
**新增：** `commitLane` 承诺机制——一旦开始挖某条队列，锁定直到目标颜色到达队头，防止中途放弃。  
**通关率：** A组 91.0%（272/299），B组 93.6%（160/171）

---

### v9 — nearUnlock（2026-04-24）

**核心逻辑：** 在轨道有车但仍 inFallback 时，若某条队列 dist=1（停一步即解锁），直接执行并设 commitLane。  
**补充场景：** v8 的 allEmpty 挖坑只在轨道完全空时触发，nearUnlock 填补了"轨道有车但仍 inFallback"的场景。  
**通关率：** A组 91.3%（273/299），B组 94.2%（161/171）

---

### v10 — 容量感知评分（2026-04-24，当前版本）

**核心思路：** 用逐格四方向遮挡深度矩阵替换原有弹药差值评分，感知每种颜色在棋盘上的真实暴露状态。

**新增函数：**
- `buildCellDepth(logic)` — 逐格计算四方向最小遮挡深度，depth=0 表示从某方向直接暴露
- `computeUrgency(cellDepth, logic)` — `urgency[color] = Σ 1/(depth+1)`，方块越浅越紧迫
- `v10Score(c, urgency, colorCount, colorAmmo, exposureMap, inFallback)` — 融合 urgency、弹药匹配度、曝光位置权重的综合评分

**评分公式：**
```
可达色:  score = urgency × (1/(1+|ammo-blocks|)) × (1/(1+ep/(TOTAL_DIST×2)))
停车候选: score = 0.6 × target_urgency × ammoFit × (1/(1+cost/20))
```

**保留机制：** exposureMap 位置权重、同色衰减（×0.6^onTrack）、commitLane、nearUnlock、bufferDanger 全部继承。  
**通关率：** A组 **92.6%（277/299）**，B组 **95.9%（164/171）**，零失败

## 下一步方向

v10 已触及贪心框架上限。剩余 29 个卡关关卡的根因是：

1. **死车封锁**（colorCount=0 的车卡住队列头，如 B62）— 需要"穿透死车"部署逻辑
2. **多步颜色依赖**（A→B→C 顺序依赖链）— 需要至少 3~5 步前向搜索
3. **容量极限**（轨道+buffer 容量不足以缓存所有中间状态）— 需要道具配合

可行改进方向：
- **BFS/DFS 前向搜索**：在部署决策时向前搜索 N 步，评估未来状态
- **死车过滤**：candidates 中过滤掉 colorCount=0 的车，而非加入竞争
